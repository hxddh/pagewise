# PageWise v3.4.2 Bug 挖掘报告

日期：2026-07-07 · 基线：`main`（170bda5，v3.4.2）。
方法：三路对抗性深挖(Rust 后端并发、跨模块单例并发、此前未覆盖的 UI/状态),每个候选都要求"先尝试用现有守卫反驳,只报存活项";作者对高危项手工二次核实。TypeScript 零错误、188 测试全绿基线(Rust 因审查环境缺 GTK 未编译,基于源码推理)。

本轮聚焦此前几轮**覆盖较浅或未触及**的区域。整体结论:并发架构比 v3.1.0 基线稳健得多,经典的"陈旧结果写错文档 / 切换竞态"绝大多数已被守卫挡住;新发现集中在**缓存新鲜度、文件系统 TOCTOU、偏好读改写、Escape 层管理**这几处。

---

## 一、确认发现(按严重度)

### R1. 中危 — PdfCache 在解析后才取 mtime,可永久缓存旧内容
`src-tauri/src/pdf.rs:208`(插入 `:217`,读回 `:171`)

`parse_all_pages` 在 `load_document`(193)和逐页提取循环(200-205,大 PDF 可耗时数秒)**之后**才于第 208 行 `file_mtime(path)`。若文件在解析期间被改写,第 208 行拿到的是**新 mtime**,而 `pages_raw` 是**旧内容**,于是缓存映射 `新mtime → 旧内容`。此后每次 `cached_pages` stat 文件、看到 mtime 相等 → **永久返回旧内容**(直到下一次改写再变 mtime)。第 212 的双检无效(插入时缓存为空)。
**修法**:在 `load_document` 之前捕获 mtime 并用它插入——这样解析期间的改写会推进真实 mtime,后续读取 mtime 不匹配 → 缓存未命中 → 重新解析(最坏一次冗余解析,绝不返回陈旧内容)。

### R2. 中危(安全) — `write_text_file` 悬空软链接绕过软链接逃逸防护
`src-tauri/src/lib.rs:244-253`

软链接逃逸检查以 `resolved.exists()` 为前提,但 `Path::exists()` **会跟随软链接**,对**悬空**软链接返回 `false` → 检查被跳过 → `fs::write`(O_CREAT)跟随软链接在授权目录**之外**创建文件。因为前端授权的是整个父目录(`save-markdown.ts` → `allowPathPersisted` 注册父目录),攻击者若能在该用户目录放文件,可预置 `notes.md` 为指向 `~/.config/autostart/x.desktop` 之类不存在路径的软链接;用户"另存为 notes.md"即在授权树外落盘。已存在文件的软链接情形已处理,**悬空是漏洞**。
**修法**:改用 `symlink_metadata`(不跟随软链接)——叶子若是软链接(悬空或否)一律拒绝;`NotFound` 视为新文件放行;普通已存在文件仍做 canonicalize 在树内校验。

### U1. 中危 — `setLocaleMode` 未加锁的读改写,会丢失并发的偏好更新
`src/i18n/index.tsx:78-86`

切换语言在 `prefsLock` **之外**读取整个偏好对象,再合并写回,会静默回退一个并发加锁写入者刚改的字段。`preferences.ts` 专门提供了在锁内读的 `patchPreferences`,而 `setLocaleMode` 不安全地重实现了它:
```ts
const current = await loadPreferences();   // 锁外读(陈旧快照)
const next = { ...current, locale: mode };
await savePreferences(next);               // 写有锁,但快照已旧
```
**失败场景**:用户在设置里几乎同时切换"默认跟随 agent"(`patchPreferences`,加锁)和语言。交错:`setLocaleMode` 读到 `followAgentDefault:true`;跟随开关提交 `false`;`setLocaleMode` 用陈旧快照写回 → `followAgentDefault:true` 复活,开关丢失。写本身是串行的(非撕裂写),问题在陈旧读快照。
**修法**:`setLocaleMode` 改调 `patchPreferences({ locale: mode })` 并 `setPrefs(result)`。

### U2. 中危 — SettingsDrawer 的 Escape 层在无关重渲染时被重新推到栈顶,导致 Escape 路由错乱
`src/components/SettingsDrawer.tsx:98-113`(对照 `DocumentSearch.tsx:33-41` 的正确写法)

推送 Escape 层的 effect 把"push 层"和"keydown 处理"耦合在一起,且依赖数组含 `requestClose`;而 `requestClose` 的身份随 `aiFooter?.dirty` 变化。于是当 AI 设置面板的防抖自动保存(或保存状态)在**另一个更上层的 overlay 打开时**翻转 `dirty` → 该 effect 重跑 → cleanup pop 掉层、push 一个**新 id**,把 Settings 推到 Escape 栈**顶**——即使命令面板在它之上。
**失败场景**:打开 Settings(层 A)→ ⌘K 打开命令面板(层 B,栈顶)→ AI 面板自动保存触发使 `dirty` 变化 → Settings 的 push effect 重跑 → 栈变成 `[B, A']`、**A' 在顶**。此时按 Escape 关掉的是底下的 Settings 而非聚焦中的命令面板。焦点仍在面板(`useFocusTrap` 不因这些重渲染重跑),两者失步。
**修法**:把 push/pop 拆成独立、依赖仅 `[open]` 的 effect(照 `DocumentSearch` 做),keydown 里从 ref 读 `requestClose`。CommandPalette/RecentFilesDrawer 同类写法建议一并统一。

## 三、对抗性核实为"无 bug"(展示覆盖面)

这些都是特意尝试攻破、最终被现有守卫挡住的路径:

- **i18n 插值对用户数据安全**:`interpolate` 用**函数**替换器,文件名含 `$&`/`$1`/`{{count}}`/`{`/`}` 都按字面插入、不二次扫描、不递归;缺失键返回键路径。复数选择 en/zh 均正确。
- **IME/输入法组合安全**:composer 和命令面板都在 Enter 前短路 `isComposing || keyCode===229`。
- **Overlay 锁在并发加锁下平衡**:`isOverlayOpen` 读 `openOwners.size`,push/pop 共享该 Set,乱序关闭仍平衡;焦点恢复按 LIFO 各自捕获 `previouslyFocused`,正确。
- **索引状态 doc A→B 切换不泄漏**:`cancelIndex` bump `pathGenerations`,每个写点都 re-check `isCurrentGeneration`+`docCache.has`,晚到结果发 idle、`upsertPageText` 对已移除路径 no-op;A→B→A 同样安全。
- **PDF 切换不渲染错文档**:`renderEpoch`+`pdfDocEpoch`/`pdfActivePath`+Rust `fileReadGen`/scope cancel 三重防御全在 `clearPdfCache` 触发;`commitLoadedDocument→clearPdfCache→setActivePdfPath` 顺序安全。
- **agent-view-context 队列 + 图片回退重试**:FIFO 深度在双发/切换中发送/图片拒绝重试下保持平衡;陈旧 ctx 被 `requireLoadedDoc` 失败兜住(非数据泄漏)。
- **generation refs 与 epochRef 不会误放行陈旧写入**:额外 bump 只会让更多回调失效(fail-safe);构造不出穿过 `streamAgentGenRef !== agentGenRef` 守卫的陈旧写。
- **监听器/订阅无泄漏**:usePageIndexStatus/useFollowAgent/docCache.subscribe/subscribeAgentProgress 均在卸载/依赖变化/finish 时退订。L3(index-events map 泄漏)已在 v3.4.1 修复,现走 `notify` 不写 map。
- **autosave vs 关窗 flush vs 切换保存**:三者都经 `saveChat(path, msgs)`、path 取自 write 时的 `documentRef.current` 并空守卫;M3 修复(flush 先 await stream-idle 再快照)完好,无写错文档/新覆旧。
- **Rust 后端**:per-scope 原子 generation(capture-before-spawn + 每步 staleness 检查)race-free;缓存锁纪律好(昂贵解析在锁外);`panic=unwind`+`catch_unwind` 覆盖所有 pdf-extract panic 路径,`spawn_blocking` 的 JoinError 全处理;页码 1-based 一致无 off-by-one;`MAX_READ_BYTES` 双重限制。

## 四、修复优先级建议

1. **快速高收益**:R1(mtime 提前捕获)、U1(setLocaleMode 改用 patchPreferences)——都是几行、明确正确。
2. **安全 & UX**:R2(悬空软链接改 `symlink_metadata` 拒绝)、U2(拆分 Escape 层 effect,统一 overlay 注册)。
3. **可选**:F1(索引用量 per-run 化,消除遥测错归因)、R3/R5 硬化、U3/U4 卫生项。

## 五、总体评价

经过三轮迭代,PageWise 的并发与切换正确性已相当扎实——本轮跨模块并发深挖只剩一个此前已知的遥测归因问题,其余全部被守卫挡住,这是很强的健壮性信号。新发现集中在两类此前未深查的角落:**缓存新鲜度/文件系统 TOCTOU**(Rust 侧 R1/R2/R3/R4)和**偏好读改写 / Escape 层生命周期**(前端 U1/U2)。其中 R1(缓存旧内容)和 U1(丢偏好)是明确的正确性 bug、R2 是可利用的安全硬化点,建议优先处理。
